# Tairitsu Framework - Platform WIT API Requirements

> Generated by hikari project based on actual usage analysis.

## Overview

This document lists WIT (WebAssembly Interface Types) APIs that consumers need from the tairitsu platform layer but are currently missing or incomplete.

### What Already Exists in tairitsu WIT

The following WIT definitions are already present in `packages/browser-worlds/wit/`:

- **dom.wit** (hand-written v0.1.0): `node`, `document`, `window`, `style` interfaces
- **browser-full.wit** (v0.2.0): Unified world with event callbacks, timer/animation callbacks, resize/mutation observer callbacks, media query callbacks, platform-helpers, console, event-target, style, lifecycle, types (dom-rect), plus auto-generated credential/WebGL/codec interfaces
- **generated/** (auto-generated v0.2.0 from W3C webref): `dom.wit` (35 interfaces), `html.wit`, `css.wit`, `events.wit`, `media.wit`, `observers.wit`, `misc.wit`, `canvas.wit`, `fetch.wit`, `storage.wit`, `url.wit`, `crypto.wit`, `performance.wit`, `web-animations.wit`, `workers.wit`, `webrtc.wit`, `device.wit`, `geolocation.wit`, `notifications.wit`, `payments.wit`, `permissions.wit`, `file-api.wit`, `streams.wit`, `svg.wit`, `indexed-db.wit`, `websockets.wit`, `wasm.wit`, `service-workers.wit`

### Gap Analysis

Despite extensive auto-generated WIT coverage, the hikari consumer (`packages/components/src/platform/wit.rs`) requires **convenience wrappers** that combine multiple low-level WIT calls into single high-level operations. These are not raw browser API mirrors but composite operations that cross interface boundaries.

## Priority P0 - Required for Component Functionality

### DOM Element Access

All of these compose `document` + `node` + `element` WIT interfaces but need single-call convenience bindings.

| Function | WIT Signature | Used By |
|----------|---------------|---------|
| `get_element_by_id` | `func(id: string) -> option<u64>` | RichTextEditor, QRCode, various |
| `query_selector` | `func(selector: string) -> option<u64>` | RichTextEditor, layout components |
| `query_selector_all` | `func(selector: string) -> list<u64>` | DragOverlay, list components |
| `element_from_point` | `func(x: s32, y: s32) -> option<u64>` | DragOverlay, context menus |
| `element_closest` | `func(element: u64, selector: string) -> option<u64>` | DragOverlay, event delegation |

**Status in WIT**: `document.get-element-by-id`, `document.query-selector`, `document.query-selector-all` exist in `dom.wit`. `element_from_point` and `element_closest` are **missing** — they require `document.elementFromPoint()` and `element.closest()` which exist in `css.wit` as part of the `element` interface but are not exposed as standalone free functions in `platform-helpers`.

### DOM Rect / Geometry

| Function | WIT Signature | Used By |
|----------|---------------|---------|
| `get_bounding_client_rect` | `func(element: u64) -> dom-rect` | Tooltips, modals, drag |
| `get_element_rect_by_id` | `func(id: string) -> option<dom-rect>` | Tooltips, popovers |
| `get_bounding_rect_by_class` | `func(class: string, element: u64) -> option<dom-rect>` | Nested component layout |

**Status in WIT**: `platform-helpers.get-bounding-client-rect` exists in `browser-full.wit`. `element.get-bounding-client-rect` exists in `css.wit`. The composite `get_element_rect_by_id` (lookup + rect) and `get_bounding_rect_by_class` (query within subtree + rect) are **missing**.

## Recommended Implementation Order

1. **Expand `platform-helpers`** in `browser-full.wit` with free-function wrappers for the most-used composite operations:
   - `get-element-by-id` (free function)
   - `element-from-point`
   - `element-closest`
   - `get-scroll-y` (free function)
   - `scroll-to`
   - `copy-to-clipboard` / `read-clipboard`
   - `request-fullscreen`

2. **Add HTMLElement-specific bindings** to the `element` interface in `css.wit`:
   - `get-inner-html` / `set-inner-html`
   - `get-content-editable` / `set-content-editable`
   - `get-outer-html` / `set-outer-html`

3. **Add media element convenience wrappers** to `platform-helpers`:
   - Free functions wrapping `html-media-element` methods
   - Audio context creation and analyser node setup

4. **Add canvas convenience wrappers** from the `Platform` trait to `platform-helpers`:
   - `get-canvas-context`
   - `canvas-set-fill-style`
   - `canvas-fill-rect`
   - `canvas-clear-rect`

5. **Add fullscreen API** — new WIT interface:
   - `request-fullscreen(element: u64)`
   - `exit-fullscreen()`
   - `get-fullscreen-element() -> option<u64>`

## Notes

- The `Platform` trait in `tairitsu_vdom::platform` (trait.rs) defines the Rust-side abstraction. The WIT definitions define the WASM boundary. These need to stay in sync.
- The `dom_ops.rs` module in `tairitsu_vdom` uses raw function pointers (`unsafe fn`) for event handler context. The WIT bindings need to support this pattern.
- All browser objects use opaque `u64` handles in the WIT layer. The Rust `ElementHandle` and `EventHandle` traits in tairitsu_vdom wrap these.
- The callback-based design in `browser-full.wit` (event-callbacks, timer-callbacks, etc.) is correct for WASI components — the host calls into the guest. Free functions like `set-timeout` take a `callback-id` rather than a closure.

## WIT Architecture Refactoring (2026-04)

Restructured how WIT interface definitions are organized in `packages/browser-worlds/wit/`:

### File layout
- `wit/handwritten/*.wit` — 4 files of framework abstractions W3C doesn't cover
  - `types.wit` (`component-types` interface — `content-editable-state` record)
  - `callbacks.wit` — 9 callback interfaces (event, timer, animation, resize-observer, etc.)
  - `component.wit` — lifecycle, console, event-target interfaces
  - `platform-helpers.wit` — constructors, callback registration, custom convenience helpers
- `wit/generated/*.wit` — 32 auto-generated per-domain WIT files from `generate_browser_wit.py`
- `wit/w3c-idl-full.wit` — auto-generated pure W3C interfaces (557 interfaces)
- `wit/browser-full.wit` — auto-generated combined file (571 interfaces = 14 handwritten + 557 W3C)

### Migrated from platform_helpers to auto-generated bindings
~19 functions in `wit_platform.rs` now use native W3C bindings:
`window::get_inner_width/height`, `element::get_bounding_client_rect/closest`,
`resize_observer::observe/unobserve/disconnect`, `mutation_observer::observe/disconnect`,
`document::element_from_point`, `base_audio_context::create_analyser`,
`audio_context::create_media_element_source`, `element::request_fullscreen`

### Rules (enforced)
- **Handwritten WIT is ONLY allowed** when W3C auto-generated interfaces are insufficient
- **`w3c-idl-full.wit`** must NOT be manually edited
- **`browser-full.wit`** is auto-generated by `generate_browser_wit.py` — do NOT manually edit
- **`event-target`** kept as handwritten (returns listener-id, not W3C addEventListener pattern)
- **`clipboard`** kept as handwritten sync wrappers (awaiting Promise→async scaffolding)

## Daemon Error Reporting Fix

The packager daemon (`packages/packager`) previously silently swallowed build errors:

### Before
- `fork_daemon()` redirected child stdout/stderr to log files immediately
- `daemonize_self()` called before initial build, so all output went to files
- Parent returned `Ok(())` immediately, printing "Daemon started" regardless of build result

### After
- `fork_daemon()` no longer redirects stdout/stderr — child keeps terminal output
- `daemonize_self()` delayed until after initial build succeeds
- Child writes readiness signal file (`tairitsu-packager.ready`) on success or error on failure
- Parent polls signal file with 5min timeout before reporting "Daemon started"
- Build failures displayed to user's terminal with `exit(1)`

## Remaining TODOs

### Promise→async/await scaffolding (medium priority)

Build helper scaffolding to convert JavaScript Promise handles (u64) to Rust async/await. Currently async W3C APIs like `clipboard.read()` / `clipboard.write()` return Promise handles but we have no way to await them from Rust.

**Blockers:**
- Need a WIT-level async abstraction or a polling mechanism
- `wasm32-wasip2` async support is still experimental

**Affected APIs (currently kept as handwritten sync wrappers):**
- `clipboard.read_text()` / `clipboard.write_text()`
- `navigator.geolocation.getCurrentPosition()`
- `fetch()` response body consumption
- `FileReader` / `FileWriter` operations
- `IndexedDB` transactions

**Proposed approach:**
1. Add a `poll-promise` WIT function: `poll_promise(promise: u64) -> option<string>` that returns `None` while pending, `Some(result)` when resolved
2. Build a Rust `async fn await_promise(promise: u64) -> String` wrapper using `async_loop { poll_promise(promise) }`
3. Migrate clipboard and other async APIs from handwritten sync wrappers to auto-generated Promise-returning bindings + the async wrapper

### Pure reference browser-full.wit (low priority, blocked on WIT tooling)

Currently `browser-full.wit` inlines all 571 interface definitions because WIT tooling (`wit-bindgen` 0.54.0) does not support `include` directives.

**Proposed approach once tooling supports it:**
1. `browser-full.wit` becomes a pure composition file with only `use`/`import`/world declarations
2. Each interface stays in its source file (`generated/*.wit` or `handwritten/*.wit`)
3. Reduces file from ~16K lines to ~600 lines

### SSR auto-generated Host trait cleanup (low priority)

The SSR crate (`packages/ssr`) now has 543 auto-generated stub interfaces from `browser-full.wit`. Most are no-ops returning `0`/`None`/`false`. Consider:
1. Selectively implementing only the interfaces actually used by SSR (e.g., `document`, `element`, `window` for viewport dimensions)
2. Reduce compilation time by not generating stubs for unused interfaces
3. Or: add `#[allow(unused)]` at the bindgen level to suppress warnings
